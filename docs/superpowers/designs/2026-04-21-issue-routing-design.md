| Label | 含義 | 對應 repo 群 |

### ⚠️ 設計關鍵:Exception Registry(維護型對照表)

由於 label → group 對應**有例外**(例:KEYPO 大部分在 `KEYPO/*` 但 `llmprojects/keypo-agent` 在另一個 group),**不能靠 naming convention 自動推斷**。系統必須:

1. 有一份 **maintained config** 存 label → {primary_group, exceptions} 的對照表
2. 例外 list 可以**隨發現逐步補齊**(初期只填 `llmprojects/keypo-agent`,日後工程師遇到 routing 不準就補)
3. 格式建議(config YAML 放在 `eng-daily-update-dashboard/config/label-routing.yaml`):
   ```yaml
   labels:
     K5:
       product: KEYPO
       primary_group: KEYPO
       known_exceptions:
         - llmprojects/keypo-agent
         # add more as discovered
     DV:
       product: DailyView
       primary_group: dailyview
       known_exceptions:
         - dv-report
         - dv-survey
     Fanti:
       product: Fanti
       primary_group: null  # intentionally cross-group
       known_repos:
         - cdp/fanti-*
         - CrawlersV2/fanti-*
         - KEYPO/fanti_*
     # ...
   ```
4. **Fanti 是特例**:本身就 by design 分散在多個 group,不算例外,算「正常的 cross-group label」
5. Runtime 行為:系統 suggest repo 時
   - 先列 primary_group 下的 repos(前幾名,依 recent activity)
   - 加上 known_exceptions(固定列在最後)
   - LLM 讀 issue 描述判斷是否特別點名 known_exceptions 裡的某一個(用來 promote 到第一名)

### Pivot Decision

- **原 P1「追 commits」死** → 被 **「靠 labels + assignees + comments」取代**
- **原 Approach B(skill factory + clustering)簡化** → 因為 **labels 本身就是 skills 的分群**,不需要額外 clustering
- **Effort 下降**:4-5 天 CC → **2-3 天 CC**

**意外的好消息**:team 95% 都乖乖貼 labels。這是多數團隊做不到的紀律。它讓系統可以更簡單、更快出貨。

## Problem Statement

公司內部 GitLab(`biglab.buygta.today`)有三個 ticket 收集站:
- `techcenter/reportcenter`(bug report)
- `techcenter/reportcenter_confidential`(機密 bug)
- `techcenter/feature-request`(feature request)

2026 Q1 共 100+ 張 issue 進入這三個站。**痛點**:

1. **Triage latency** — IC 不會主動監看 inbox,被提醒才撥空處理
2. **Evaluation fatigue** — 被提醒後,判斷「這該改哪個 repo、解法是什麼」仍要花 15-30 分鐘,因為要翻 closed issue、問 senior、或硬著頭皮讀 code
3. **Status drift** — IC 一忙就忘了回覆 ticket status,CSM(Ivy Ma 代表)端等待時間過長,無法向客戶 commit timeline
4. **Tribal knowledge 集中** — senior IC 頭腦中的 pattern(「這類 issue 通常改 X repo」)無法被複用,junior 重跑同樣的摸索

**系統目標**:新 issue 進來 2 分鐘內,把「相似 closed issue + 可能的 repo + 初步 plan 摘要」推送到 Google Chat 現有 daily-update space,讓 IC 一眼決定能否 claim,降低 CSM 端的 status drift 與 IC 端的 evaluation overhead。

## Status Quo

### 一張新 `reportcenter` issue 今天早上掉進 inbox 後的 10 分鐘

根據 user 描述(逐句):

> 「IC 被提醒, 才會撥空確認 issue/requirement, 有過長的 time lag, 而且評估方案 可能時間過久, 或者一忙就忘了回覆 ticket status」

關鍵訊息:
- **IC 不會主動 pull** — 等 CSM / 管理者 / 系統提醒才動
- 被提醒後,IC 的行為**不可預測**:「可能翻 / 可能不翻 closed issue / 可能問 senior」
- Status 的回覆也是**非主動**的,容易在忙碌中斷線

### 「IC 會自己翻 closed issues 嗎?」的診斷結果

答案是 **有時候**,沒有穩定模式。這個結論對設計極為重要:**不能假設 IC 會主動使用 dashboard view**。系統必須在**不依賴 IC pull** 的情況下產生價值 — 也就是 push-based。

### 當前流程的時間成本(推估,待量測)

| 階段 | 平均時間 | 主要成本 |
|---|---|---|
| Issue 進 inbox → 有人注意 | 可能數小時到 1 天 | 依賴外部提醒 |
| 注意 → 開始 triage | 10-60 分鐘 | IC 當下是否能切換 task |
| Triage → 分派給某 IC | 5-30 分鐘 | Team lead / manager 介入 |
| IC 接手 → 釐清「這是什麼問題、改哪個 repo」 | 15-45 分鐘 | **評估方案時間過久** — 系統主要要攻的點 |
| IC 處理完 → 回覆 status | **經常拖延或遺失** | 這是另一個 status drift 點 |

**總 lead time**:easily 4-24 小時才進入「真正動工」。系統若能把最後兩階段加速 5-10x,一週替團隊省 5-10 小時。

## Target User & Narrowest Wedge

### Primary User
**工程 IC(8-14 人團隊)**,被提醒後決定「這我接不接、該怎麼做」的那個人。

### Secondary User
**Ivy Ma(CSM senior manager)**,需要看到 triage 狀態以便回覆客戶時程。

### Beneficiary
**User 本人(工程主管)** — 省下親自分派 + 追蹤 status 的時間。

### Narrowest Wedge(已確定 scope)

**資料範圍**:只做 `reportcenter` + `reportcenter_confidential`。`feature-request` 驗證後再加(user 明確決定)。

**功能範圍(S1')**:

```
新 issue 進 reportcenter / reportcenter_confidential
    ↓
[Auto-triage batch job, <2 min]
    ├─ 找 top 3 最相似 closed issues(+ 它們的 cross-ref commits)
    ├─ 抽取「通常改哪幾個 repo」signal
    ├─ 用 Claude 生 5 句摘要 + 初步 plan draft
    ↓
Post 到 Google Chat space `AAQAQhmoRAk`(既有 daily update space)
    ├─ 格式:issue link + 摘要 + 建議 repos + 相似案例 + plan 3-5 行
    ├─ **不要 approval gate** — 僅 info
    ├─ IC 在 Chat thread 用 reaction 或 reply 表態(自發,不強制)
    ↓
Dashboard 同步加一個 view(歷史回查,非主入口)
```

**刻意排除的功能**(v1 不做):
- Approval loop(等 IC 點「OK」才 commit) — 會重建 status drift
- 自動 assign IC — 政治敏感,v1 不碰
- 自動 close issue — 系統不做決策
- 跨系統 webhook(Zendesk、Intercom 等外部 ticket 源) — `feature-request` 後再想

## Constraints

1. **技術**:必須掛在現有 `eng-daily-update-dashboard` 的 pipeline。不能另起獨立服務(user 不想養第二套)。
2. **資料**:所有 issue + commit cross-reference 來自 `biglab.buygta.today` GitLab API,PAT scope `read_api` 已有。
3. **身分**:memberMap 必須可重用現有 `gitlab-config.json`、`github-config.json`、`chat-config.json` 的映射。
4. **延遲**:analysis 完成到 Chat post 2 分鐘內。5 分鐘以上就失去價值。
5. **機密**:`reportcenter_confidential` 的內容不可 leak 到非機密 space。需 space-level 隔離(可能要第二個 Chat space)。
6. **成本**:Claude API 成本每週控制在 < $5(100+ issue/Q ÷ 13 週 ≈ 8/週,每個約 $0.05-0.10)。

## Premises

以下前提已與 user 確認(P1 經 revision,P2-P6 以 fast path `rest agree`):

### P1 (revised by user)
```
Closed issues 可以透過 GitLab cross-reference 機制,回溯到修它的 commits。
從 commits 擷取:
  - 改了哪些 repos
  - 改了哪些 files
  - commit message 的解法摘要
  - related merge request 的 discussion

這些資訊是系統的 "fix trace",比人寫的 resolution note 可靠。
```

**驗證條件**(Pre-build):
- 隨機抽樣 20 筆最近 closed `reportcenter` issues
- 數出有幾筆在 GitLab 上有 `Linked items` / `Related commits`
- **門檻:≥ 12/20(60%)**
- 低於 60% 代表 cross-reference 習慣不夠普及,需:
  - 選項 A:教育團隊補 `Closes #123` 習慣(治本)
  - 選項 B:改用 LLM 擷取 issue discussion 的 free-text 作為 fallback(補破網)

### P2 — Skill-factory 架構 > runtime LLM
Auto-generated skill files(可 version control、可 review、可手調)> runtime LLM over full corpus(快但不透明、不可改)。

### P3 — ICs 會讀 Chat post
Ivy + eng team 已在 space、習慣已建立。**未驗證**:會不會**點進去看** vs 划過去。**這就是 spike 要驗證的事**。

### P4 — v1 不做 approval gate
Post = info,不等 ack。Approval loop 會重建 status drift 問題,v2 視反饋再加。

### P5 — 100+/Q = 足夠訓練 pattern,但單 repo 樣本少
要**跨 repo 泛化**(「這類 issue 通常改 billing-service **或** user-portal」),不要過度 fit 單一 repo。

### P6 — 分析速度 > 完整度
70% 對的分析 2 分鐘出來 > 100% 對的 30 分鐘後出來。允許系統說「我不確定」,讓 IC 用原方式處理。

## Approaches Considered

### Approach A: Rules + LLM summarize(MVP / Spike)
- 手寫 5-10 條路由規則(keyword → 建議 repo)
- LLM 讀 issue 生 5 句摘要
- keyword 搜尋 top-5 相似 closed issues
- Post 到 Chat **test space**(先不動 prod space)

**Effort**:human 1 週 / CC+gstack 1-2 天
**Risk**:低
**Completeness**:6/10 — 無學習,規則表會變維護地獄
**Reuses**:`prepare-task-analysis.js` pattern、`chat-config.json`、既有 gitlab-commits.js 的 API client

### Approach B: Skill factory(推薦終態)
- Batch job:把每個 closed issue + cross-ref commits 組合成「case」
- LLM 驅動 clustering,相似 case 歸一群
- 每群自動產一個 gstack 風格 skill 檔(`.md` with YAML frontmatter):
  ```yaml
  ---
  name: issue-type-payment-failure
  description: 付款失敗類型的 ticket
  trigger_patterns: ["付款", "payment", "交易失敗", "500 error /api/pay"]
  typical_repos: [billing-service, user-portal]
  common_files: [src/payment/*.ts]
  sample_cases: [#123, #456, #789]
  solution_template: |
    1. 確認 payment_logs 錶
    2. ...
  ---
  ```
- Skills 存在 repo(例如 `docs/issue-skills/`),可 review、可 PR、可手調
- Runtime:新 issue → LLM 看 skill descriptions 選 best match → 執行 skill 的 solution_template → 生 plan
- Post 到 Chat

**Effort**:human 3 週 / CC+gstack 4-5 天
**Risk**:中 — clustering quality 決定 skill 品質
**Completeness**:9/10 — 對齊 user mental model、可複用、可演進
**Reuses**:同 A + 加一個 skill generator script

### Approach C: RAG over commits(lateral)
- 不做 clustering、不產 skill 檔
- Embed 每個 commit message(跨所有 repo)
- 新 issue → semantic search top-k commits → 抽 repo + 改過的 files → LLM 生 plan

**Effort**:human 4 週 / CC+gstack 1 週
**Risk**:中高 — embedding 基礎設施 + 可解釋性差
**Completeness**:7/10 — 繞過 issue 資料品質問題,但失去「issue → solution」narrative,IC 看不懂為什麼系統建議這個 repo
**Reuses**:commit fetch script,但需新增 embedding 服務

### Rejected:S3(獨立系統)
Webhook-driven、不掛 dashboard,user 明確拒絕(「不想養第二套」)。

## Recommended Approach

### **Approach B(終態)+ Approach A 當 2 天 Spike**

執行順序:

#### Phase 1(Day 1-2):Spike = Approach A prototype

**目標**:驗證 P3(IC 會讀 Chat post)

步驟:
1. 手寫 5-10 條最明顯的路由規則(例:「付款」→ billing-service、「登入」→ auth-service)
2. 寫 `scripts/prepare-issue-analysis.js`,對稱於 `prepare-task-analysis.js`
3. LLM 生 5 句摘要 + 建議 repo
4. Post 到**新建的 Chat test space**(不動 prod `AAQAQhmoRAk`,避免影響 daily update 閱讀體驗)
5. 邀請 **3 位 IC + Ivy + user 本人**加入 test space
6. **跑 2 週**:每個新 issue 都 post 進去

**驗證指標**:
- [ ] 有 ≥ 1 位 IC 在 thread 用 reaction 或 reply(代表有讀且思考)
- [ ] Ivy 有 ≥ 1 次 reply 問後續 status
- [ ] User 本人發現花在 manual triage 的時間減少

**失敗定義**:2 週後,Chat test space 完全冷清、沒人理。→ **砍掉,不進 Phase 2**。Save 2-3 週工時。

#### Phase 2(Week 3-5):Approach B 正式

**前提**:Phase 1 成功,P3 驗證

步驟:
1. 寫 `scripts/build-issue-skills.js`:跑 batch clustering,產 `docs/issue-skills/*.md`
2. 團隊 review 前 20 個 skill(人工品管)
3. 寫 runtime matcher(`scripts/match-issue-skill.js`):新 issue → LLM 選 skill → 執行 template
4. 把 Phase 1 的 Chat integration 改接到 Approach B
5. Dashboard 加一個 view `IssueRouting.tsx`,展示歷史 analysis(非主入口)
6. 逐步 roll out:reportcenter → reportcenter_confidential(2 週觀察期)

## Open Questions

以下問題**不必在 start 之前全答**,但要在 spike → build 之間釐清:

1. **P1 驗證結果**:cross-reference 覆蓋率是否 ≥ 60%?
2. **Clustering 粒度**:5 個大 skill 還是 30 個細 skill?影響 precision vs recall,可用 elbow method 決定。
3. **Confidential isolation**:機密 issue 的 analysis 能否 post 到同一個 Chat space?(建議 Phase 2 建第二個 space 專給 confidential)
4. **手調 skill 的 review process**:誰 review、多久一次?
5. **Cost control**:每週 Claude API 支出上限?目前估 < $5,但 skill clustering 的 batch job 會一次性燒較多。
6. **Status feedback loop**:v2 加 approval gate 會解 status drift,還是再破壞?要不要改成 "IC 動工時 reply 🟡、修完 reply 🟢" 的輕量協定?

## The Assignment

**在寫 code 前,user 必須完成這一件事(30 分鐘)**:

1. 打開 `https://biglab.buygta.today/techcenter/reportcenter/-/issues?state=closed`
2. 隨機點 **20 筆**最近關閉的 issue
3. 每筆記一下:
   - GitLab 頁面右側有沒有 `Linked items` 或 `Related merge requests` 自動顯示?
   - 如果有,點進去確認 merge request 的 source project(也就是「這個 issue 修在哪個 repo」)
4. 統計:**20 筆中有幾筆能從 GitLab 自動 cross-reference 回到修復 repo**

**若 ≥ 12 筆(60%)**:P1 驗證通過,直接開 Phase 1 spike
**若 < 12 筆**:先不做 — 教育團隊寫 `Closes #` 兩週,再來評估

**為什麼要先做這個**:這是整個系統的地基。如果 cross-reference 覆蓋率不夠,clustering 沒 signal,skill 品質會垃圾,下游再精巧都沒用。30 分鐘的檢查省下 5 週的 wrong direction。

## What I noticed about how you think

1. **你跳過了兩次「資料品質檢查」問題,但最後 P1 的 revision 比我寫的還漂亮** — 你沒有 verify 資料,但你知道**怎麼繞過資料品質問題**(用 cross-reference 取代 human-written resolution note)。這是**架構直覺**,不是嚴謹 validation。兩種技能都有用,但你要知道自己是哪一種。未來 build 時記得補 validation 這一環,因為架構直覺在**新 domain** 會失準。

2. **你看到 existing tool 時沒有自豪地展示,是被我追問才帶出來** — 然後一次性給完整 context(reportcenter/confidential/feature-request 三個 repo、staging 策略)。這代表**你不愛推銷既有成就**,但當情境需要時能立刻 map 出來。對 diagnostic 合作來說這是好 trait,但對 team pitch 或 VP 溝通是劣勢 — 很多 stakeholder 需要聽到你過去做過什麼才會 buy in 你下個提案。

3. **你選 scope 的 pattern 是「先做 1 & 2,驗證後做 3」** — 這是正確的 intrapreneur 直覺。但注意:**我提出 S1/S2/S3 時,你沒選功能 scope,你選了資料 scope**。兩個是不同軸。這可能意味你腦內的 default 是「先做完整功能、但只在子集資料上」,而不是「先做最小功能、在全資料上」。這個 default 在 intrapreneurship 通常 OK(你有 political cover),但創業情境會讓你做太多、ship 太慢。

4. **當我挑戰「closed issue 是 skill 的 search corpus」時,你立刻 reframe 成 training data** — 這是**快速接住挑戰並 upgrade 自己 mental model** 的能力。很少人會這樣,多數人要嘛死守原觀點要嘛完全倒向挑戰者。你走了中間路線:「你的 challenge 合理,但我要換一個更好的框架容納它」。這個 pattern 在 founder 身上很珍貴。

Hi [名字],

我正在規劃一個**內部工程效率工具**:新進的 GitLab issue(`biglab.buygta.today/techcenter/reportcenter` 與 `reportcenter_confidential`)會透過 Anthropic Claude API 做自動分析,產出 routing + plan draft,協助 IC 快速 triage。詳細設計 doc 可提供。

在開始實作前,有一個合規問題**需要您確認**才能進行。

### 資料流

```
GitLab issue(含 title / description / comments)
    ↓
本地 node script(內部伺服器)
    ↓  [HTTPS, 加密傳輸]
Anthropic Messages API(Claude Sonnet 4.6)
    ↓
API 回應(結構化 JSON)→ 本地 SQLite 暫存 → Google Chat post
```

### 涉及的資料類型

- Issue title / description(可能含客戶回報的 bug 細節)
- Issue labels / assignee(內部人員)
- 過去相似 closed issue 的摘要(每個 500 字以內)

**特別值得注意**:`reportcenter_confidential` 的 issue 可能含客戶組織名、聯絡人、交易資料等敏感內容。

### 我需要的確認

1. **公司與 Anthropic 是否有 DPA(Data Processing Agreement)或等效的資料處理合約?**
2. 如果有:**該合約是否涵蓋「內容含客戶資料的 issue」傳入 API 做分析**?
3. 如果沒有:**是否要先簽署 DPA**,或是否要將此用途限縮在 non-confidential issue(`reportcenter` 不含 `_confidential`)?

### 技術面可提供的佐證

- **資料最小化**:只傳送 issue 分析所需欄位,不傳送整個 GitLab repo 內容
- **保留**:Anthropic API 預設不將 input 用於訓練(可提供他們的 commercial terms 連結)
- **本地儲存**:API response 存在公司內部伺服器 SQLite,不外流
- **可關閉開關**:`allow_confidential_llm=false` 的 config flag 可隨時關閉 confidential 走 API 這條路徑,退回 rule-based 路由

實作計畫本週準備好,但**這題未確認前不會動手**。預計等您回覆後決定:
- A. 全綠燈 → confidential + non-confidential 全走 LLM
- B. 部分綠燈 → 只 non-confidential 走 LLM,confidential 走 rule-based
- C. 需要先補 DPA → 我等

方便今明兩天回一下嗎?或我們開 10 分鐘會議討論?

## T3 — Slack DM 或 email 給 GitLab Admin / DevOps

**To:** [DevOps / GitLab 管理員]
**Subject:** GitLab PAT scope 升級申請 — `read_api` → `api`(for issue routing 系統)

---

Hi [名字],

快速請求:我需要把現在 `eng-daily-update-dashboard` 專案用的 GitLab PAT,從 `read_api` 升級到 `api` scope。

### 原因

新的 issue routing 系統(design doc + implementation plan 可提供)需要**寫 comment 回 issue** — IC 在 Google Chat approve 分析結果後,系統把 plan 貼回原 GitLab issue 作為 note,讓 CSM 能從 issue 看到處理方向。

### 現在的 scope vs 需要的 scope

| | `read_api` (目前) | `api` (需要) |
|---|---|---|
| 讀 issue / MR / project | ✅ | ✅ |
| 讀 notes | ✅ | ✅ |
| **寫 issue note** | ❌ | ✅ |
| 改 issue 狀態 / labels | ❌(不使用) | 不會使用 |
| 管理 repo / user | ❌(不使用) | 不會使用 |

系統**只會寫 note**,不會改 state / labels / assignees。這點可以在 code review 時驗證 — 所有 API call 集中在 `lib/gitlab-client.mjs`,writer methods 只有 `postIssueComment` 一個。

### Security / rotate policy 提案

- **Scope**: `api`(最小必要)
- **Lifetime**: 90 天 rotate 一次
- **Storage**: 本地 server 的 `.env`,mode 0600,不進 repo
- **Audit**: 每次 write 都 log issue_uid + timestamp + IC approval signature,保留 90 天
- **Revoke plan**: 任何時候 InfoSec 判定要 revoke,system 會 fallback 到「Chat post 但跳過 GitLab comment」模式,不會 crash

### 請求

1. 產一個新的 `api`-scope PAT,expiration 90 天
2. 傳給我(secure channel,不要 Slack plain text)
3. 通知我 rotate 日期,我到時再申請新的

系統的 secrets management 我會遵現有 pattern。

謝謝,

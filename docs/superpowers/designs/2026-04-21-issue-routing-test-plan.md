## What this system does(給 QA tester 的一句話)

新 issue 進到 GitLab `techcenter/reportcenter` 或 `reportcenter_confidential` → 系統每 15 分鐘掃 → 用 Claude Sonnet 4.6 做 routing + plan draft → post 到 Google Chat space(card + approve/edit/dismiss buttons)→ IC 按 Approve → 系統把 plan 作為 comment 貼回 GitLab issue。

## Surface to Test

### 1. Data input — GitLab issue events

| Event | Expected | Why it matters |
|---|---|---|
| 新 issue 建立(labels: K5, P1_高) | 2 分鐘內 Chat post,含建議 repo + plan draft | 主流程 |
| Issue labels 變動(K5 → K5+Bug) | Chat threaded reply「🔄 Update: routing 更新」 | P1.5 dedup 邏輯 |

| Issue reopened | 重新分析 post | 邊界 |

### 2. Chat interaction — IC 在 Google Chat 的動作

| Interaction | Expected | Page/Space |
|---|---|---|
| 按 ✅ Approve button | GitLab issue 出現 comment(內含 plan);Chat card 更新為「Approved by {IC} at {time}」 | `spaces/AAQAQhmoRAk` thread |
| 按 ✏️ Edit button | Dialog 跳出,IC 編輯 plan,submit 後 GitLab comment 含編輯版 | 同上 |
| 按 ❌ Dismiss button | 無 GitLab comment;Chat card 更新為「Dismissed by {IC}」 | 同上 |
| IC 按 Approve 後,另一 IC 再按 | 顯示「已 approved by X @ Y」,不重複 post comment | 同上 |
| 同一 IC 快速雙擊 Approve | Idempotent — 只 post 一次 comment | 同上 |
| 未在 memberMap 的人按 button | 拒絕操作,顯示「非授權成員」 | 同上 |

### 3. Webhook security — Apps Script doPost 當 endpoint

| 場景 | Expected |
|---|---|
| 合法 Google Chat payload(含 signature) | pass,dispatch action |
| Forged payload(無 signature / signature 錯) | 401,log 來源 IP |
| Replay attack(複用舊 request) | 檢查 timestamp,> 5 分鐘拒絕 |

### 4. GitLab output — comment 貼回 issue

| 場景 | Expected |
|---|---|
| Approve 成功 | GitLab issue 出現 comment,作者顯示 system bot,內容含 plan + 歸因「由 /issue-routing 系統建議,IC {name} approved」 |
| GitLab 401 / token revoked | Alert 給 user 本人,Chat card 更新「post 失敗,請手動」 |
| GitLab 429 rate limit | Retry 3 次 with backoff(5s / 30s / 2min) |
| Issue 已被 close | 跳過 post,Chat card 更新「issue 已關,不再 post」 |

## Critical Paths(必測 E2E)

1. **Happy path routing**
   - Fixture:K5+P1_高 issue with 「agent 回覆出錯」 description
   - Expected:Chat post 建議 `llmprojects/keypo-agent`(LLM 應判讀為 agent issue),plan draft 有 3-5 步

2. **Happy path approval → GitLab comment**
   - 接續上面 → IC 按 Approve → GitLab issue 出現 comment
   - Expected:comment 格式正確、有 system 歸因、有 IC approved signature

3. **Label 改動 re-analyze**
   - Issue labels 從 K5 改成 BD
   - Expected:threaded reply,新的 repo 建議(bigdata/* 取代 KEYPO/*)

4. **Issue closed 流程**
   - Open issue → post → closed → final reply
   - Expected:threaded 最終 reply 含 closer + closed_at

5. **Confidential issue 走完整 flow**(allow_confidential_llm=true)
   - Issue 在 `reportcenter_confidential` → 完整 LLM 分析 + post
   - Expected:同 happy path,但 Chat space 可能需 isolation(見 Open Questions)

## Edge Cases(必測 unit / integration)

### Data edge cases
- **Zero similar issues(cold start)** — 新 label 從沒出現過,系統 skip Phase 2,post 精簡 Chat「無歷史可參考」
- **Label 不在 `label-routing.yaml`** — Fallback 到純 assignee-frequency,Chat post 註記「未知 label,建議補 config」
- **Issue 無 labels** — Post 「無 label,無法路由」,建議 IC 補 label 後 system 會自動 re-analyze
- **Issue description 超長(> 8K chars)** — 截取前 2K + 最後 1K,LLM 收到 summary 版

### Concurrency edge cases
- **兩個 cron 併發** — File-level `flock`,後到的 skip
- **Webhook 同時多個 button click** — SQLite transaction 確保 approval_status 狀態機 atomic
- **LLM call 超時(>60s)** — 中斷,標記 issue `analysis_failed`,下輪 cron retry

### Failure cascade edge cases
- **LLM JSON 格式錯誤(schema validation fail)** — 三次重試,全失敗 fallback 到 rule-based 路由 + Chat post 註記「LLM 分析失敗,僅提供 label 路由」
- **SQLite corruption / lock timeout** — Alert,下輪 cron 前修復
- **Apps Script webhook cold start(>5s)** — Google Chat 會 retry;state 要 idempotent 處理

### UX edge cases
- **Plan draft 含敏感字串**(e.g. 客戶真名)— Phase 2 prompt 要明示 LLM「不 echo 原 description 的 PII」
- **Chat post 超過字數上限(4096 chars)** — 分段 post,或截斷加「完整版見 thread reply」

## Eval Test Suite(CRITICAL — prompt change gate)

**Golden fixture**:20 筆 closed issue 快照,含原始 metadata + 人工標註的「正確路由」+「合理 plan 範例」

### Cases(依照實際 label 分佈)
- 10 筆 K5(6 個 infra、4 個 agent)
- 4 筆 Fanti(1 crawler、1 backend、1 UI、1 integration)
- 2 筆 BD
- 2 筆 DV
- 2 筆 無已知 label / unsure

### Assertions per fixture
- `layer` 輸出正確(對 Fanti)或 "n/a"(對其他)
- `suggested_repos[0]` 在預期 repo set 裡(allow top-3 任一個算對)
- `suggested_assignees` 含歷史 top-2 assignee
- `confidence` 對 unsure case < 0.5,對 clear case ≥ 0.7
- `plan_draft` 非 null(對 high-conf)/ null(對 low-conf)
- Output JSON 通過 schema validation

### Baselines
- 首次跑 → record 作為 v0 baseline
- Prompt 改動 → 跑 eval,比對 v0
- **Gate**:≥18/20 pass,不然 prompt 改動不准 merge

### Frequency
- 每次 prompt / schema / config 改 → 必跑
- 每月 1 日 → 定期跑(catch model drift)
- 新 model version upgrade → 必跑

## Security & Compliance Tests

- **Confidential data flow**:assert `allow_confidential_llm=false` 時 `reportcenter_confidential` issue 不呼叫 Anthropic API(用 network intercept 驗證)
- **PAT scope 最小原則**:GitLab PAT 新增 `api` scope 僅供 comment post,不會被用於其他操作(code review:所有 API call 都在 `gitlab-client.mjs`,可 audit)
- **Chat webhook signature**:拒絕 forged payload(hmac verification test)
- **SQLite 權限**:`db/issue-routing.sqlite` mode 0600,只有 dashboard process 可讀寫

## Open Questions(給 `/qa` 階段回來確認)

1. **Chat space isolation**:confidential issue 要不要 post 到**另一個 Chat space**(避免 non-confidential 成員看到)?design doc 只提 v2 考慮,要先跟 InfoSec 確認當前 space 成員名單 vs confidential 授權名單
2. **PAT 權限升級流程**:誰申請 `api` scope PAT?Rotate 頻率?Revoke 機制?
3. **Eval fixture 來源**:20 筆 golden issues 要**匿名化**嗎?(若 fixture 進 repo 被 commit)
4. **Approval 超時**:Chat card 送出後 IC 24 小時沒點任何 button,系統要不要**auto-escalate**(tag Ivy / manager)?
5. **GitLab comment 格式 i18n**:全中文 or 中英混?客戶看得到的場景下要不要避免技術 jargon(repo name、commit hash)?
